第四章 系统设计

系统设计是将需求转化为具体技术实现蓝图的关键过程。本章将立足于 ACIR “沉浸式心流”与“数据驱动”的核心理念，从总体架构、数据库模型以及界面交互（UI/UX）三个维度，详细阐述系统的设计方案。

4.1 总体架构设计

为了实现高可用、可扩展且前后端解耦的系统目标，ACIR 采用了基于 B/S（Browser/Server）模式的现代 Web 分层架构。系统整体划分为前端展示层、后端服务层与数据持久层，各层之间通过标准协议进行通信。

4.1.1 逻辑架构设计
系统逻辑架构自上而下分为四层：
（1）前端展示层（Client Layer）：
该层是用户与系统交互的直接界面，包括 PC 端浏览器与移动端 H5 页面。基于 Vue 3 框架构建，采用 SPA（单页应用）模式。前端通过 Axios 库封装 HTTP 请求，与后端 RESTful API 进行数据交互；利用 Pinia 进行全局状态管理（如番茄钟计时状态、用户 Token）；使用 ECharts 实现复杂的数据可视化渲染。
（2）网关与路由层（Gateway/Nginx）：
在生产部署中，Nginx 作为反向代理服务器，负责静态资源（HTML/CSS/JS）的托管与 API 请求的转发。它同时承担了跨域处理（CORS）、Gzip 压缩以及 SSL 安全加密的职责，确保数据传输的高效与安全。
（3）后端服务层（Service Layer）：
该层基于 Spring Boot 3 框架构建，遵循 MVC 设计模式。Controller 层负责接收 HTTP 请求并进行参数校验；Service 层封装核心业务逻辑（如番茄钟状态流转、任务看板排序算法）；DAO 层（Data Access Object）通过 MyBatis 与数据库交互。此外，该层还集成了 Google Gemini SDK，负责处理与大语言模型的流式对话逻辑。
（4）数据持久层（Data Layer）：
选用 MySQL 8.0 作为关系型数据库，存储用户信息、任务详情及专注流水记录。对于热力图等复杂统计需求，利用 MySQL 的聚合函数与索引优化查询性能。

4.1.2 物理部署架构
在开发与测试阶段，系统采用前后端分离的本地部署模式。前端 Vite 服务器运行在 5173 端口，后端 Spring Boot Tomcat 容器运行在 8080 端口，MySQL 数据库运行在 3306 端口。前后端通过 API 代理解决跨域问题。在云端部署时，推荐使用 Docker 容器化技术，将前端 Nginx、后端 Jar 包与 MySQL 镜像编排在同一虚拟网络中，以实现环境的一致性与快速交付。

4.2 数据库设计

数据库设计是系统稳定运行的基石。ACIR 的数据模型围绕“用户-任务-专注”这一核心业务闭环展开，共设计了 5 张核心实体表。

4.2.1 概念模型设计（E-R图）
系统的实体关系（Entity-Relationship）如下：
- 用户（User）：系统的核心主体，拥有唯一的 ID、用户名与加密密码。
- 任务（Task）：属于特定用户，包含标题、状态、优先级等属性。一个用户可以拥有多个任务（1:N）。
- 标签（Tag）：用于对任务进行分类，包含名称与颜色。一个任务可以拥有多个标签，一个标签也可以关联多个任务（N:M），通过中间表 `task_tags` 实现关联。
- 子任务（Subtask）：依附于父任务，用于细化工作内容。一个任务可以包含多个子任务（1:N）。
- 专注记录（PomodoroRecord）：记录用户的一次完整或中断的专注过程。一条记录必须属于一个用户，且可以（可选）关联到一个具体任务（1:N）。

4.3 界面与交互设计 (UI/UX)

ACIR 的界面设计不仅仅是为了美观，更是为了服务于“心流（Flow）”这一心理学状态。系统采用了“Glassmorphism（毛玻璃）”作为核心设计语言，结合动态背景与响应式布局，旨在减少视觉噪点，引导用户进入深度工作状态。

4.3.1 设计理念：Immersive Flow（沉浸式心流）
传统效率工具往往采用类似 Excel 的密集表格布局，这种设计虽然信息密度高，但容易给用户带来压迫感。ACIR 提出了“Immersive Flow”设计理念：
- **视觉减负**：大量使用半透明背景与模糊效果（`backdrop-filter: blur(20px)`），使界面元素仿佛悬浮于壁纸之上，营造轻盈通透的质感。
- **内容聚焦**：首页（Focus Station）摒弃了复杂的仪表盘，仅保留一个巨大的 Hero Timer（计时器）与当前聚焦的任务。当用户开始计时后，鼠标静止 3 秒，所有无关 UI（导航栏、按钮）将自动淡出，实现“零干扰”体验。
- **动态氛围**：系统支持根据用户上传的背景图片，利用 `fast-average-color` 算法自动提取主色调，并智能调整文字颜色（深/浅）与遮罩透明度，确保在任何壁纸下文字都清晰可读且视觉和谐。

4.3.2 布局演进与重构
为了适应从桌面端到移动端的跨设备访问，系统的导航布局经历了三次迭代：
- **v1.0 侧边栏模式**：采用传统的左侧固定侧边栏（Aside Sidebar）。优点是导航层级清晰，缺点是占据了宝贵的横向空间，且在移动端难以适配。
- **v2.0 顶部双 Header 模式**：桌面端改为顶部导航。为了兼顾沉浸感与功能性，设计了“双 Header”机制：
    - **Hero Header**：绝对定位，背景透明，仅在页面顶部显示，与 Hero 背景融为一体。
    - **Fixed Header**：固定定位，带有毛玻璃背景。当用户向下滚动页面超过 Hero 区域时，该 Header 会自动滑出，确保导航功能随时可用。
- **v3.0 移动端 Bottom TabBar 模式**：针对移动端（视口 < 768px），彻底隐藏顶部导航，转而采用底部的标签栏（TabBar）。这符合拇指操作的热区习惯，并且通过 CSS `safe-area-inset-bottom` 适配了现代全面屏手机的底部手势条。

4.3.3 核心界面交互设计
（1）首页（Focus Station）：
采用“Hero Section + Bento Grid”布局。上半部分为全屏高度的计时器区域，支持番茄钟/休息模式切换与 Zen Note 快速记录；下半部分为响应式的 Bento Grid（便当盒）布局，展示热力图概览、今日任务列表等微件。在移动端，Bento Grid 自动降级为单列流式布局。

[此处插入图 4-3：首页 Focus Station 界面设计图（PC端与移动端对比）]

（2）任务看板（Kanban Board）：
采用横向滚动的多列布局。每个任务被设计为一个卡片（Card），包含标题、优先级标签与子任务进度条。交互上，利用 `Sortablejs` 实现了流畅的拖拽体验：用户可以按住卡片，将其从“待办”列拖入“进行中”列，或在同一列内上下拖拽调整优先级。拖拽过程中，系统提供实时的占位符（Ghost Element）反馈，确保操作的确定性。

[此处插入图 4-4：任务看板拖拽交互设计图]

（3）智能助手（Intelligent Assistant）：
采用类似 IM（即时通讯）的对话窗口布局。左侧为 AI 回复气泡，右侧为用户提问气泡。输入框固定在底部。为了增强沉浸感，聊天背景同样应用了毛玻璃效果，使对话内容与全局壁纸产生视觉联系。

4.4 本章小结
本章从宏观架构到微观交互，系统地阐述了 ACIR 的设计方案。在架构上，确立了 Spring Boot + Vue 的前后端分离模式；在数据上，设计了以 User-Task-Pomodoro 为核心的 E-R 模型；在交互上，确立了 Glassmorphism 视觉风格与响应式导航策略。这些设计决策紧密围绕“提升效率”与“优化体验”的需求目标，为下一章的系统实现提供了详尽的蓝图。
